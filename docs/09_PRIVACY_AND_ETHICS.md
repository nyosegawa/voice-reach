# 09 --- プライバシー・法的・倫理的設計

## 1. プライバシーアーキテクチャ

### 1.1 ローカルファースト原則

VoiceReachは**ローカルファースト**を最上位のプライバシー原則とする。生体データ・カメラ映像・個人識別情報はローカルに保持し、クラウドには抽象化・匿名化されたテキストデータのみを送信する。

VLMによる環境認識を含む全パイプラインにおいて、この原則を一貫して適用する（レポートB5 Q5）。

```
┌─────────────────────────────────────────────────────────────┐
│               患者端末（Mac mini M4 Pro、ローカル）            │
│                                                             │
│  ▌永久ローカル保持 --- 外部送信禁止                          │
│   ├ 内向きカメラ映像の生データ                                │
│   ├ 外向きカメラ映像の生データ                                │
│   ├ 顔画像・顔特徴量（MediaPipe Face Mesh 468点）            │
│   ├ 視線追跡の生時系列データ（MobileOne s0 + L2CS-Net）       │
│   ├ 表情・感情検出の生データ（4ch: 瞳孔径/rPPG/視線/まばたき）│
│   ├ rPPG心拍推定の生データ（open-rppg）                      │
│   ├ 音声録音の生データ                                       │
│   ├ 指センサーの生データ（PZT + FSR）                        │
│   └ パーソナルベースラインモデル                              │
│                                                             │
│  ▌ローカル処理 --- テキスト化後のみ送信可                    │
│   ├ VLMシーン記述（MiniCPM-V 4.0 ローカル推論結果）          │
│   │    → テキスト化済み環境コンテキストのみクラウドへ         │
│   ├ 感情推定サマリ（テキスト化済みスコアのみ）               │
│   └ 音声認識テキスト（話者名は匿名化）                       │
│                                                             │
│  ▌ローカル保持 --- 暗号化バックアップ可                      │
│   ├ PVP（Personal Voice Profile）                            │
│   ├ 会話ログ                                                │
│   ├ 音声クローンモデル（CosyVoice 2/3）                     │
│   └ 登録済み顔データベース                                   │
│                                                             │
└──────────────────────┬──────────────────────────────────────┘
                       │ HTTPS（TLS 1.3）
                       │ 送信データ（すべてテキスト、匿名化済み）:
                       │  - PVP（構造化テキスト、~2000トークン、実名不含）
                       │  - コンテキストフレーム（テキスト化済み環境・状態記述）
                       │  - 会話履歴（直近10ターンのテキスト）
                       ↓
┌─────────────────────────────────────────────────────────────┐
│               クラウドLLM API                                 │
│  プライマリ: Gemini 2.5 Flash（ステートレス推論）             │
│  セカンダリ: Claude Haiku 4.5（フォールバック）               │
│  ターシャリ: GPT-4.1 nano（緊急フォールバック）               │
│                                                             │
│  - リクエストごとにデータ破棄（ゼロリテンションポリシー）     │
│  - ログに個人識別情報を記録しない                             │
│  - 学習データとして使用しない（全プロバイダーAPI利用規約確認済）│
└─────────────────────────────────────────────────────────────┘
```

### 1.2 データ分類と保護レベル

| 分類 | データ例 | 保存場所 | 暗号化 | アクセス権 |
|---|---|---|---|---|
| 最高機密 | カメラ映像（内向き/外向き）、生体生データ、VLM入力画像 | ローカルのみ | AES-256 | 患者本人のみ |
| 高機密 | PVP、会話ログ、音声クローンモデル、顔登録DB | ローカル+暗号化バックアップ | AES-256 | 患者+指定家族 |
| 中機密 | 状態サマリ、介護者向けレポート、VLMテキスト出力 | ローカル+介護者共有 | TLS転送時 | 介護チーム |
| 低機密 | システム設定、UIプリファレンス | ローカル+クラウド同期 | TLS転送時 | 患者+管理者 |

### 1.3 カメラ・センサー別のデータ処理ポリシー

VoiceReachはデュアルカメラ構成（内向き/外向き）を採用する。各カメラの映像データに対し、以下の処理ポリシーを適用する。

#### 内向きカメラ（患者の目・顔）

```
ポリシー: 生体データは一切クラウドに送信しない

処理パイプライン:
  Webカメラ映像（30fps, 720p）
    ↓ すべてローカル処理
    ├→ MediaPipe Face Mesh（468点ランドマーク）  → ローカルのみ
    ├→ MobileOne s0 + L2CS-Net → 視線座標         → ローカルのみ
    ├→ open-rppg → 心拍推定                       → ローカルのみ
    ├→ 瞳孔径変動 → 感情チャネル                  → ローカルのみ
    └→ EAR → まばたき検出                         → ローカルのみ

クラウドへの送信: 一切なし
  ※ テキスト化された感情サマリ（例: "valence: 0.1, arousal: 0.4"）
     のみコンテキストフレームの一部としてクラウドLLMに送信される。
     映像・画像・生体生データは送信しない。
```

#### 外向きカメラ（周囲環境）

```
ポリシー: 原則ローカル処理。クラウド送信時は匿名化+明示的同意

3層ハイブリッド処理（レポートB5 5.1節）:

  Layer 1: OpenCV動体検出（常時、CPU ~5%）     → ローカルのみ
  Layer 2: MiniCPM-V 4.0（4.1B, ローカルVLM）  → ローカルのみ ★
    ├→ 定期スナップショット（5-10秒間隔）
    ├→ テキスト化されたシーン記述 → コンテキスト統合層へ
    └→ 画像データは推論後即破棄（数秒以内）
  Layer 3: Gemini 2.5 Flash（クラウドVLM）      → 患者同意時のみ
    ├→ 送信前に匿名化処理（顔ぼかし、個人情報マスク）
    ├→ 患者の明示的同意を毎回取得（またはセッション単位で設定可能）
    └→ APIゼロリテンションポリシー確認済み

★ MiniCPM-V 4.0のローカル処理により、通常時は映像データが
   デバイスの外に出ることは一切ない。
   これがVoiceReachのプライバシー設計の最大の利点の一つである。
```

**VLMローカル処理のプライバシー利点**（レポートB5 Q5）:
- MiniCPM-V 4.0（4.1B）はApple Silicon M4 Pro上で~45 tok/s、メモリ~2.5GBで十分実用的に動作する
- 映像データがデバイス外に出ないため、ネットワーク傍受・クラウド侵害・プロバイダーによるデータ利用のリスクをゼロにできる
- ALS患者のベッドサイドという極めてプライバシー性の高い環境（身体介助中の映像等）において、ローカル処理は患者の尊厳を保護するために必須である
- オフライン環境でも環境認識が継続動作する（設計書01で規定する70%以上のオフライン動作要件を満たす）

### 1.4 クラウドLLM送信データの仕様

クラウドLLM API（Gemini 2.5 Flash / Claude Haiku 4.5 / GPT-4.1 nano）への送信データを厳密に規定する（レポートB2 9.2節）。

```
送信が許可されるデータ（すべてテキスト形式）:
  1. PVP（~2000トークン）
     - 患者の表現パターン・語彙・語調の構造化記述
     - 実名は含めない（PVP内でも「患者」と表記）
     - 対話相手の名前はファーストネームのみ（同意取得済み）

  2. コンテキストフレーム（~500トークン）
     - VLMのテキスト出力（"妻がベッドサイドに来ている"等）
     - 環境データ（室温、時刻、スケジュール等）
     - 感情推定サマリ（数値スコアのみ、生データではない）

  3. 会話履歴（~500トークン、直近10ターン）
     - テキスト化された発話ログ
     - 話者名はファーストネームのみ

  4. タスク指示（~300トークン）
     - 候補生成ルール、意図軸分散指示

送信が禁止されるデータ:
  ✗ カメラ映像・画像データ（内向き/外向き共に）
  ✗ 顔画像・顔特徴量
  ✗ 視線追跡の生時系列データ
  ✗ 生体信号の生データ（rPPG波形、瞳孔径時系列等）
  ✗ 音声録音の生データ
  ✗ 指センサーの生データ
  ✗ 患者のフルネーム・住所・医療ID等の個人識別情報
```

### 1.5 オンデバイス処理の最大化

全パイプラインをMac mini M4 Pro（24GB）上のMLXフレームワークで実行する。VLMを含めたローカル処理の範囲は以下の通り（レポートB5, B6）。

```
オンデバイス処理（ローカル完結）:
  ✅ 視線推定（MediaPipe + MobileOne s0 + L2CS-Net, ~8ms）
  ✅ 表情追跡（MediaPipe Face Mesh 468点）
  ✅ まばたき検出（EAR）
  ✅ rPPG心拍推定（open-rppg）
  ✅ 指入力処理（PZT + FSR, 4層フィルタリング）
  ✅ VLM環境認識（MiniCPM-V 4.0, 4.1B, ~45 tok/s） ← 新規追加
  ✅ PVP抽出（Qwen3-1.7B, ローカルLLM）
  ✅ 暫定候補生成 Stage 1（Qwen3-0.6B, ~525 tok/s）
  ✅ 改善候補生成 Stage 2（Qwen3-1.7B, ~320 tok/s）
  ✅ 音声合成（CosyVoice 2/3, ストリーミング150ms）

クラウド利用（オプション、同意時のみ）:
  ☁️ 高品質候補生成 Stage 3（Gemini 2.5 Flash / Claude Haiku 4.5）
  ☁️ VLMフォールバック（Gemini 2.5 Flash、ローカルVLMで判断困難時）

全パイプライン同時実行メモリ: ~9-11GB / 24GB（十分な余裕）
```

## 2. 法規制への準拠設計

### 2.1 GDPR（EU一般データ保護規則）への準拠

VoiceReachがEU圏内の患者に使用される可能性を考慮し、GDPR準拠を設計段階から組み込む。

| GDPR原則 | VoiceReachの対応 |
|---|---|
| **合法性・公正性・透明性**（第5条1(a)） | 項目別インフォームドコンセント。処理目的の明示。AI候補生成の仕組みを説明 |
| **目的限定**（第5条1(b)） | データはAAC候補生成の目的のみに使用。マーケティング・第三者提供に使用しない |
| **データ最小化**（第5条1(c)） | クラウドにはPVP+コンテキストの匿名化テキストのみ送信。生体・映像データは送信しない |
| **正確性**（第5条1(d)） | PVPの内容を患者/家族が確認・修正可能 |
| **保存制限**（第5条1(e)） | クラウドLLMはステートレス（データ保持なし）。ローカルデータは患者が削除可能 |
| **完全性・機密性**（第5条1(f)） | ローカルAES-256暗号化。転送時TLS 1.3。VLMプロセスのネットワークアクセスをデフォルト遮断 |
| **アクセス権**（第15条） | 患者は全データの閲覧・エクスポートが可能 |
| **削除権（忘れられる権利）**（第17条） | 全データの不可逆削除が可能。クラウド側はステートレスのため削除不要 |
| **データポータビリティ**（第20条） | PVP・会話ログを標準形式（JSON）でエクスポート可能 |
| **プロファイリングに対する権利**（第22条） | AI候補はあくまで提案。最終選択は常に患者本人が行う |
| **DPO（データ保護責任者）**（第37条） | 医療データを扱うため、DPOの選任が必要 |
| **DPIA（データ保護影響評価）**（第35条） | 高リスク処理（生体データ+プロファイリング）のため、DPIA実施が義務 |

### 2.2 日本の個人情報保護法への準拠

| 法的要件 | VoiceReachの対応 |
|---|---|
| **要配慮個人情報**（第2条第3項） | カメラ映像・生体データ・病歴関連データは要配慮個人情報に該当。取得にあたり本人の同意を取得 |
| **利用目的の特定と通知**（第17条・第21条） | 「AAC候補生成のための表現パターン分析」等、具体的な利用目的を明示 |
| **第三者提供の制限**（第27条） | クラウドLLM APIへの送信は「委託」として整理。匿名化テキストのみ送信し、プロバイダーとのDPA（データ処理契約）を締結 |
| **外国にある第三者への提供**（第28条） | Gemini（Google）・Claude（Anthropic）・GPT（OpenAI）は米国企業。APPI第28条に基づく措置（相当措置の確認、本人同意）を実施 |
| **安全管理措置**（第23条） | ローカル暗号化（AES-256）、アクセス制御、通信暗号化（TLS 1.3）、操作ログ記録 |
| **個人情報保護委員会への報告**（第26条） | 漏洩時の報告体制を事前に構築。72時間以内の報告義務に対応 |
| **医療情報ガイドライン**（厚労省 3省2ガイドライン） | 医療情報を扱うシステムとして、3省2ガイドラインに準拠した安全管理措置を実装 |

### 2.3 プロバイダー別データ保護ポリシーの確認

クラウドLLM APIプロバイダーのデータ取り扱いポリシーを確認し、VoiceReachの要件を満たすことを検証する（レポートB2 9.2節）。

| プロバイダー | API経由データの学習利用 | データ保持期間 | ゼロリテンション | DPA対応 |
|---|---|---|---|---|
| Google（Gemini 2.5 Flash） | API経由はデフォルトで不使用 | リクエスト処理後に破棄 | Vertex AI経由で保証 | 対応 |
| Anthropic（Claude Haiku 4.5） | デフォルトでオプトアウト | リクエスト処理後に破棄 | 商用利用時に保証 | 対応 |
| OpenAI（GPT-4.1 nano） | API経由はデフォルトで不使用 | リクエスト処理後に破棄 | Enterprise版で保証 | 対応 |

## 3. 同意とデータ制御

### 3.1 インフォームドコンセント

```
同意が必要な項目（項目別に個別同意）:
  1. 内向きカメラによる常時顔撮影（視線・表情・生体データ取得）
  2. 外向きカメラによる環境撮影（VLMシーン認識）
  3. マイクによる常時音声モニタリング
  4. 過去のメール/SNS/音声からのPVP抽出
  5. 顔認識のための家族・介護者の顔登録
  6. クラウドLLMへのコンテキスト送信（匿名化テキストのみ）
  7. クラウドVLMへの画像送信（ローカルVLM判断困難時のフォールバック）
  8. 介護者へのパッシブ検出情報の共有
  9. 音声クローンの作成と使用
  10. 医師向けレポートの自動生成

同意の取得方法:
  - 初期セットアップ時に一括ではなく、項目別に同意
  - いつでも撤回可能（設定画面から）
  - 意思決定能力がある段階で取得
  - 視線入力での同意操作も可能にする
  - クラウドVLM利用（項目7）は特に慎重に:
    セッション単位の包括同意 or イベントごとの個別同意を選択可能
```

### 3.2 データソース別の制御

患者は各データソースの使用範囲を細かく制御できる。

```
設定例:
  Gmail:
    ✅ 家族へのメール → PVPに使用
    ✅ 友人へのメール → PVPに使用
    ❌ 仕事のメール → PVPに使用しない
    ❌ 特定の相手のメール → 除外

  LINE:
    ✅ 妻とのトーク → PVPに使用
    ✅ 家族グループ → PVPに使用
    ❌ 特定のトークルーム → 除外

  カメラ:
    ✅ 外向きカメラ → VLMローカル処理ON
    ⚠️ 外向きカメラ → クラウドVLMフォールバック（同意設定に依存）
    ✅ 内向きカメラ → 視線・表情解析ON
    ❌ 特定時間帯のカメラ → 一時停止設定可能

  出力制御:
    ✅ 家族への発話 → 全PVP適用
    ⚠️ 介護者への発話 → 丁寧スタイルのみ
    ❌ 特定のトピック → 候補に含めない
```

### 3.3 データ削除権

```
患者（または法的代理人）はいつでも以下を要求できる:
  - 特定のデータソースの削除
  - PVP全体のリセット
  - 会話ログの削除（全部 or 期間指定）
  - 音声録音の削除
  - 顔登録データの削除
  - VLMのローカルLoRAモデルの削除
  - アカウント全体の削除

削除は不可逆的に実行し、暗号化バックアップからも消去する。
クラウドLLM/VLMにはデータが残らない（ステートレス）ため、
クラウド側の削除は不要。

GDPR第17条（忘れられる権利）および
日本の個人情報保護法第33条（保有個人データの消去）に準拠。
```

## 4. ネットワークレベルの技術的保護措置

### 4.1 VLMプロセスのネットワーク隔離

映像データの意図しないクラウド送信を防ぐため、OS/ファイアウォールレベルでの制御を実装する（レポートB5 5.5節）。

```
VLMプロセス（MiniCPM-V 4.0）のネットワークアクセス:
  - デフォルト: 完全遮断（ネットワークアクセス不可）
  - VLMプロセスはサンドボックス内で実行
  - 映像データはプロセス間通信（IPC）でのみ受け渡し
  - 推論結果（テキスト）のみが統合エンジンに渡される

クラウドVLMフォールバック時:
  - 別プロセスとして起動
  - 送信前に匿名化パイプライン（顔ぼかし、テキストマスク）を必ず通過
  - 患者同意フラグがONの場合のみネットワーク送信を許可
  - 送信ログを記録（いつ、何枚、どのAPIに送信したか）
```

### 4.2 暗号化とアクセス制御

| レイヤー | 技術 | 対象 |
|---|---|---|
| ストレージ暗号化 | AES-256（macOS FileVault） | ローカル全データ |
| 通信暗号化 | TLS 1.3 | クラウドAPI通信 |
| アプリケーション暗号化 | AES-256-GCM | PVP、会話ログ、顔DB |
| 鍵管理 | macOS Keychain / Secure Enclave | 暗号化鍵の保護 |
| アクセス制御 | 生体認証（介護者端末）+ パスワード | 設定変更・データ閲覧 |

### 4.3 監査ログ

```
記録対象:
  - クラウドAPIへの全送信リクエスト（タイムスタンプ、送信先、データサイズ）
  - クラウドVLM利用時の画像送信イベント（匿名化処理の実行確認を含む）
  - データ閲覧・エクスポート・削除の操作
  - 同意設定の変更
  - 介護者・家族によるアクセス

保存期間: ローカルに2年間保持（法的要件に応じて延長可能）
閲覧権限: 患者本人 + 法的代理人
```

## 5. Federated LoRA によるプライバシー保護型モデル適応

### 5.1 FedVLM戦略

VoiceReachでは、患者のベッドサイド環境に特化したVLM適応を、プライバシーを保持したまま実現するために**Federated LoRA（FedVLM）**を採用する（レポートB5 Q5, 文献[9]）。

```
Federated LoRA の仕組み:

  患者端末A                    患者端末B
  ┌──────────────┐            ┌──────────────┐
  │ MiniCPM-V 4.0│            │ MiniCPM-V 4.0│
  │ + LoRAアダプタ│            │ + LoRAアダプタ│
  │              │            │              │
  │ ローカルデータ │            │ ローカルデータ │
  │ で学習        │            │ で学習        │
  └──────┬───────┘            └──────┬───────┘
         │ LoRA重みの差分のみ送信       │
         │ （生データは送信しない）       │
         ↓                             ↓
  ┌────────────────────────────────────────┐
  │          集約サーバー                   │
  │  pLoRA（個人化LoRA）の集約・配布        │
  │  差分プライバシー（ノイズ注入）適用      │
  │  個人情報の逆推定を防止                 │
  └────────────────────────────────────────┘
```

**プライバシー保護の仕組み**:
- 生データ（映像・生体情報）は一切サーバーに送信しない
- 送信されるのはLoRAアダプタの重み更新（数MB）のみ
- 差分プライバシー（Differential Privacy）を適用し、重み更新からの個人情報逆推定を防止
- 各患者のデータは完全にローカルに留まる

**VoiceReachへの適用**:
- 医療機器、介護用品、ベッドサイド家具等のドメイン特化認識精度を向上
- 患者固有の環境（部屋のレイアウト、常用品等）に適応
- 複数患者のモデル改善を集約し、全体の認識品質を向上
- 参加は完全任意。参加しなくても基本モデルで十分に動作

### 5.2 LoRAファインチューニングのローカル実行

```
ファインチューニング環境:
  - フレームワーク: MLX（Apple公式、LoRA/QLoRA対応）
  - 対象モデル: MiniCPM-V 4.0（4.1B）
  - 学習データ: 患者のベッドサイド環境の匿名化済みキャプション
  - 学習時間: 数十分（M4 Pro、数百サンプル）
  - メモリ使用: QLoRAで~4GB

ファインチューニングデータの管理:
  - 学習データは暗号化してローカルに保存
  - 学習完了後、元画像データは削除可能（LoRAアダプタのみ保持）
  - 患者の同意に基づき、LoRA重みのみをFederated集約に参加可能
```

## 6. 法的論点

### 6.1 AI生成候補と「本人の意思」

VoiceReachの核心的な法的課題は、LLMが生成した候補を患者が選択した場合、それがどこまで「本人の意思表示」として認められるかである。

```
意思表示のスペクトラム:

完全に本人の意思                    完全にAIの意思
├────────────────────────────────────┤
│          │          │             │
自由文字   候補選択    候補修正     パッシブ
入力       (明示的)   なし選択    検出のみ
│          │          │             │
確実に     おそらく    グレー       困難
本人の     本人の     ゾーン
意思       意思
```

#### 候補選択が本人の意思と認められるための条件

```
1. 候補の多様性が確保されていること
   - 同じ方向の候補ばかりでは誘導になる
   - 意図軸分散がここでも重要

2. 拒否/修正の手段が常に利用可能であること
   - 「自分で入力する」オプションが常にある
   - 候補を全て拒否できる

3. 操作ログが残っていること
   - どの候補を見て、どれを選んだか
   - 何巡目で選んだか（即座に選んだ ≈ 強い意志）
   - 感情データとの整合性

4. システムの透明性
   - 候補がどう生成されたか説明可能
   - PVPの内容を本人/家族が確認・修正可能
```

### 6.2 法的に重要な場面での特別措置

```
延命治療の意思決定:
  - AI候補選択のみでは不十分
  - 自由文字入力での明示的な意思表示を求める
  - 複数回の確認（日を置いて再確認）
  - 医師・家族の立会い
  - 操作ログの詳細記録と保存
  - 可能なら公証人の関与

遺言:
  - 自筆証書遺言の代替としての法的有効性は未確立
  - 公正証書遺言の口授の代替として認められる可能性
  - 法務省・裁判所との事前確認が必要
  - 弁護士との連携体制を構築

財産管理の指示:
  - 候補選択による指示は補助的な参考情報として扱う
  - 法的拘束力のある指示は別途手続きが必要
```

### 6.3 認知機能低下への対応

ALS患者の約15〜20%にFTD（前頭側頭型認知症）の合併が報告されている。

```
リスク:
  - 認知機能低下により候補の意味を理解できなくなる
  - 不適切な候補を無意識に選択してしまう
  - 衝動的な選択が増える

対策:
  1. 認知機能スクリーニングの定期実施
     - 操作パターンの変化を自動検出
     - 選択の一貫性スコアを算出
     - 異常を検出したら家族・医師に通知

  2. セーフガードモード
     - 認知機能低下が疑われる場合に有効化
     - 重要な意思決定には家族の承認を要求
     - 候補の複雑度を下げる
     - 不可逆な操作（データ削除等）に追加確認

  3. 段階的な代理決定への移行
     - 本人の意思決定能力の評価を定期的に実施
     - 能力に応じた権限の段階的移行
     - 法的代理人の関与レベルを明確化
```

## 7. 「その人らしさ」のAI生成に関する倫理的境界

### 7.1 根本的な問い

```
Q: AIが「その人らしい」候補を生成し、患者がそれを選んだ場合、
   その発話は「本人の言葉」なのか「AIの言葉」なのか？

考察:
  - 健常者も自分の言葉を「自分で全て創造」しているわけではない
  - 言語は社会的に共有された道具であり、誰もが借用している
  - 候補を「選ぶ」行為自体が意思の表明である
  - PVPは本人の過去の表現から構築されたものである
  - ただし、AIのバイアスが混入する可能性は否定できない
```

### 7.2 設計上の倫理ガイドライン

```
1. 透明性の確保
   - 対話相手が「AIシステムを通じた発話である」ことを認識できる状態を維持
   - ただし、毎回明示する必要はない（初回説明で十分）
   - VLMによる環境認識が候補生成に影響していることも説明可能にする

2. 操作防止
   - 候補生成にシステム提供者の意図を混入させない
   - 広告や特定の思想の誘導に使わない
   - PVPにないパターンを意図的に挿入しない
   - VLMの認識結果を恣意的にフィルタリングしない

3. 尊厳の保護
   - 患者を「AIの入出力装置」として扱わない
   - 患者が候補に依存しすぎないよう、自由入力を常に提供
   - 「その人らしさ」の定義権は患者本人にある
   - カメラ映像のローカル処理は患者の尊厳保護の技術的基盤

4. 進化の余地
   - 患者の価値観や好みは変化しうる
   - PVPを「固定されたペルソナ」としてではなく
     「進化する参照点」として扱う
   - 患者が明示的に「こう変えたい」と言ったら反映する
```

## 8. 第三者のプライバシー

### 8.1 対話相手のデータ

```
対話相手（介護者、家族、訪問者）の:
  - 顔データ: ローカルのみ保持、本人の同意で登録
  - 発話テキスト: コンテキストとして一時的に使用、長期保存は選択制
  - 映像: VLMでローカル処理後に即破棄、保存しない
  - クラウドVLMフォールバック時: 顔をぼかした上で送信（同意取得済みの場合のみ）

対話相手の権利:
  - 顔登録の拒否権
  - 自分の発話ログの閲覧権
  - 自分の発話ログの削除権
  - 録音の停止要求権
  - クラウドVLMへの自分の映り込み送信の拒否権
```

### 8.2 PVP構築時の第三者データ

```
メール/LINE等から抽出する際:
  - 相手の発言内容はPVPに含めない（患者の表現パターンのみ抽出）
  - 相手の名前はPVP内で使用（対人スタイルの切替に必要）
  - 相手が希望すれば、その人とのやりとりをPVP構築から除外可能
```

### 8.3 外向きカメラに映る第三者

```
VLM処理における第三者保護:
  - 外向きカメラに映る第三者（面会者等）の映像はローカル処理のみ
  - VLMのテキスト出力には個人を特定する情報を含めない
    （例: "男性が入室" とは記述するが、顔特徴の詳細記述はしない）
  - クラウドVLMフォールバック時は顔検出+ぼかし処理を必ず適用
  - 面会者には外向きカメラの存在と用途を掲示で通知
```
